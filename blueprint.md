# 项目：AI 高效自动化软件工厂 (Blueprint v2.2.43)

> **版本优化说明：** 根据当前 Cursor 工具版本（2.2.43）及支持的模型（参考 ![image](image.PNG)），调整工作机制与模型划分策略，以确保兼容性与操作简化。

## 1. 核心模型能力与划分

| **模型名称**         | **能力重点**             | **最佳使用场景**                       |
|----------------------|-------------------------|---------------------------------------|
| **Opus 4.5**         | 顶层架构设计支持         | 系统级分析、模块决策与复杂重构框架设计。 |
| **GPT-5.1 Codex Max**| 高效代码生成与问题修复   | 大规模代码编写、复杂问题处理与优化操作。  |
| **Gemini 3 Flash**    | 快速响应的多模态支持     | UI 分析、文档概要生成与低成本任务处理。   |
| **Sonnet 4.5**        | 专项高级推理能力         | 特定逻辑单元推导和性能优化辅助手段。      |

> ⚙️ 调用策略：根据任务复杂度，优先分配至性价比最高的模型。Opus 4.5 用于架构设计，GPT-5.1 Codex Max 用于编码主任务，Gemini 3 Flash 辅助 UI 与文档任务。

---

## 2. 协同规则 (Supervision Protocol)

### 更新后的核心规则
1. **任务调用智能分级：** Cursor 必须基于提供的模型目录（见 ![image1](image1)）智能分配工作任务。
2. **单元测试要求：** 任意代码变更均需通过 `npm test` 单元测试，覆盖率不得少于 80%。
3. **上下文解析机制：** MCP 协议升级为 2.2.43 专属版本，支持任务间上下文动态拼接。
4. **回展自愈逻辑：** CI/CD 异常自动重试 3 次并生成详细反馈文档。

> 🛠️ **操作建议：** 若 Cursor 版本过低的功能受限，请按照文档提供流程进行手动适配任务。

---

## 3. 新验收流程与目标

**3.1 验收指南**
- **接口合规性：** API 文档必须符合 OpenAPI v3.1。性能标准需控制在 150ms 响应内。
- **回归测试覆盖率：** Pull Request 只可通过测试覆盖率 >85%的情况合并。
- **安全与可用性：** 代码需通过 OWASP 静态安全审计，不得存在高危漏洞。

**3.2 提交与部署要求**
- **提交方法：** 采用语义化 Commit 格式，所有变更需要符合 PR 策略并通过 GitHub Actions。
- **部署方式：** 使用 Dockerfile 进行容器打包，通过 Cursor 自动化生成预览 URL。

---

## 4. 基础操作指引

按照版本兼容性，执行如下步骤确保工作流程清晰：
1. **环境配置**
   - **GitHub 仓库：** 创建私有库并配置必需的 Webhooks；
   - **Cursor 设置：** 配置启用模型（参考 ![image1](image1）），确保各项 API Key 激活正常。

2. **规则固化与验证**
   - 将以上规则粘贴至 Cursor 的 Settings > Rules for AI 中；
   - 使用 Composer 模式进行规则前置验证：保证模型可用性与 API 连通性。

3. **发起任务与迭代**
   - 使用 Composer 启动核心工程任务；
   - 设置动态工作目标，例如：“完成模块设计并生成验证报告”。

4. **部署与验收**
   - 自动生成的演示环境需通过目标验收标准；
   - 交付相关功能并动态调试。
